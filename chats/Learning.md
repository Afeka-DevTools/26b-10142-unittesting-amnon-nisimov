היי! בשמחה. בדיקות יחידה (Unit Tests) הן אחד מהכלים החשובים ביותר בארגז הכלים של מפתח, והן קריטיות לכתיבת קוד אמין וניתן לתחזוקה.
בדיקת יחידה מתמקדת בחלק הקטן ביותר של הקוד שניתן לבדוק באופן הגיוני – לרוב מדובר במתודה (Method) או מחלקה (Class) בודדת ב-Java. המטרה היא לקחת את ה"יחידה" הזו, לנתק אותה משאר המערכת (למשל ממסדי נתונים, רשת או מחלקות מורכבות אחרות), ולוודא שהיא מחזירה את התוצאה המדויקת שאנחנו מצפים לה עבור קלט מסוים.


איך בנויה בדיקת יחידה?
הסטנדרט המקובל לכתיבת בדיקות נקרא תבנית AAA, המחלקת כל בדיקה לשלושה שלבים ברורים:
Arrange (הכנה): יצירת האובייקטים, הגדרת המשתנים והכנת סביבת הבדיקה.
Act (פעולה): קריאה למתודה הספציפית שאותה רוצים לבדוק, ושמירת התוצאה שלה.
Assert (אימות): שימוש בפונקציות מיוחדות כדי לוודא שהתוצאה שהתקבלה בפועל (Actual) תואמת לתוצאה לה ציפינו (Expected).
הכלים המרכזיים ב-Java
JUnit: הספריה הסטנדרטית והנפוצה ביותר ב-Java לכתיבת בדיקות (כיום הסטנדרט הוא JUnit 5 / Jupiter). היא מספקת את האנוטציות (כמו @Test) ואת פעולות האימות (כמו assertEquals).
Mockito: ספריה שמשתמשים בה כדי "לזייף" (Mock) אובייקטים אחרים שהמחלקה שלנו תלויה בהם, כדי שנוכל לבדוק אותה בבידוד מוחלט.
דוגמה מעשית
נניח שיש לנו מחלקה פשוטה שמבצעת פעולות מתמטיות:
Java
public class MathUtils {
    public int add(int a, int b) {
        return a + b;
    }
}

כך תיראה בדיקת היחידה שלה באמצעות JUnit:
Java
import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.assertEquals;

public class MathUtilsTest {

    @Test
    public void testAdd_PositiveNumbers_ReturnsSum() {
        // Arrange
        MathUtils mathUtils = new MathUtils();
        int a = 5;
        int b = 3;

        // Act
        int result = mathUtils.add(a, b);

        // Assert
        assertEquals(8, result, "5 + 3 should equal 8");
    }
}

הרצת הבדיקות בסביבת הפיתוח
חלק מהכוח של בדיקות יחידה הוא הפידבק המיידי. כשעובדים בסביבות עריכה קלות כמו VS Code (עם חבילת ההרחבות של Java), הממשק מזהה אוטומטית את האנוטציה @Test. מעל כל מתודת בדיקה יופיע קישור קטן של "Run Test" או "Debug Test". לחיצה עליו תריץ את הבדיקה מאחורי הקלעים ותציג לך וי ירוק (הצלחה) או איקס אדום (כישלון) ישירות בתוך העורך, מה שמאפשר לבדוק את הקוד תוך כדי הכתיבה שלו מבלי לקמפל פרויקט שלם בכל פעם.
האם יש מחלקה ספציפית או מתודה שכתבת ואתה מנסה להבין איך לכתוב לה בדיקה עכשיו?
בקוד שכתבת מה זה MathUnils? זה מחלקה שקשורה לjunit?
לא, MathUtils היא לא מחלקה ששייכת ל-JUnit.
זוהי מחלקה שאנחנו המפתחים כותבים כחלק מהקוד הרגיל של האפליקציה שלנו (מה שנקרא Production Code). השם שלה הוא קיצור של "Math Utilities" (כלי עזר למתמטיקה), והיא שימשה בדוגמה פשוט כמחלקה רגילה שמכילה לוגיקה שרוצים לבדוק (במקרה הזה, מתודה שמחברת שני מספרים).
אז מה ההבדל ואיך הן קשורות?
ההפרדה בין הקוד של האפליקציה לקוד הבדיקה היא קריטית:
קוד האפליקציה (MathUtils): קוד עצמאי לחלוטין. הוא לא מכיר את JUnit, לא תלוי בו, ותפקידו רק לבצע את הלוגיקה של התוכנה (למשל חישובים, גישה למסד נתונים, או אלגוריתמים).
קוד הבדיקה (MathUtilsTest): זו המחלקה שבה אנחנו משתמשים ב-JUnit. היא מייבאת את הכלים של JUnit (כמו @Test או assertEquals). בתוך מחלקת הבדיקה הזו, אנחנו יוצרים אובייקט של MathUtils (הקוד שלנו), מפעילים אותו, ובודקים אם הוא מחזיר את התוצאה הנכונה.
במילים פשוטות: MathUtils הוא הקוד שרוצים לבדוק, ו-JUnit הוא כלי העבודה שבאמצעותו אנחנו בודקים אותו.

אתה צודק לחלוטין ברמת הלוגיקה – אם תשנה את הפרמטר הראשון (התוצאה המצופה - Expected) ל-7, הבדיקה אכן תיכשל (ב-Java תזרק שגיאה שנקראת AssertionError ויופיע איקס אדום בעורך הקוד, במקום רק החזרת false).
אבל, חשוב לדייק את המטרה של תהליך הבדיקה:
השאיפה שלנו היא לא לשחק עם המספרים ב-assertEquals בשביל לראות כישלונות (למרות שזו אחלה דרך לוודא שהבדיקה באמת בודקת משהו!). קוד הבדיקה צריך תמיד לייצג את "האמת המוחלטת". מכיוון ש-5 ועוד 3 תמיד שווים 8, הפרמטר הראשון בבדיקה הזו יישאר לנצח 8.
אז מתי הבדיקה באמת אמורה להיכשל?
הבדיקה נועדה להיכשל אך ורק כאשר מישהו מכניס באג בקוד האפליקציה שלך (למשל במחלקת MathUtils), ולא כשמשנים את הבדיקה עצמה.
תאר לעצמך שאתה כותב לוגיקה לתוכנה שסורקת ובודקת מבחנים של תלמידים במתמטיקה. יש לך מתודה שאמורה לשקלל את הציון הסופי של המבחן, ואתה כותב לה בדיקת יחידה שמוודאת שעבור סדרת טעויות ספציפית, הציון חייב להיות 85. הערך ב-assertEquals שלך יהיה תמיד 85.
אם בעוד חודשיים תחליט לעשות רפקטורינג (שיפור לקוד) או להוסיף פיצ'ר חדש, ותעשה בטעות קטנה בלוגיקת חישוב הציון – ברגע שתלחץ על Run Tests ב-VS Code, הבדיקה פתאום תיכשל! המערכת תתריע לך: "היי, ציפיתי לקבל 85 על סמך הבדיקה שכתבת בעבר, אבל הקוד המעודכן שלך החזיר עכשיו 90".
בצורה הזו, בדיקות יחידה הופכות לרשת ביטחון אוטומטית שרצה ברקע ומוודאת ששינויים חדשים לא שוברים קוד ישן שכבר עבד בצורה תקינה.
